# API Usage

All endpoints are available at your server root — `http://localhost:8000` locally, or your Cloud Run URL in production. Swagger UI is auto-generated at `/docs`.

---

## Endpoints

| Method | Path | Description |
|---|---|---|
| `GET` | `/health` | Liveness probe — returns status + cache size |
| `GET` | `/` | Web UI (`static/index.html`) |
| `POST` | `/query` | Synchronous multi-question Q&A |
| `GET` | `/stream-query` | SSE streaming single-question Q&A |
| `GET` | `/cache-status` | Inspect in-memory document cache |
| `POST` | `/reset` | Clear in-memory document cache |

---

## GET /health

```bash
curl http://localhost:8000/health
```

```json
{
  "status": "ok",
  "version": "2.1.0",
  "cached_document_sets": 2
}
```

`cached_document_sets` shows how many unique document sets currently have FAISS indexes held in RAM.

## POST /query

Synchronous endpoint. Processes multiple documents and multiple questions in one call. Returns all answers when the full pipeline completes.

**Best for:** scripts, batch processing, backend integrations.

### Request

```json
{
  "documents": [
    "https://example.com/policy.pdf"
  ],
  "questions": [
    "Is cataract surgery covered for 5L sum insured?",
    "What is the waiting period for pre-existing diseases?",
    "Is AYUSH treatment included?"
  ]
}
```

| Field | Type | Required | Notes |
|---|---|---|---|
| `documents` | `string[]` | ✅ | PDF URLs or local file paths |
| `questions` | `string[]` | ✅ | One or more questions |

The document pipeline runs once per unique document set (order-independent). Repeated calls with the same URLs hit the in-memory cache and skip PDF download, extraction, chunking, embedding, and FAISS build entirely.

### Response

```json
{
  "answers": [
    "Cataract surgery is covered up to ₹25,000 for a 5L sum insured.",
    "Pre-existing diseases have a 48-month waiting period.",
    "AYUSH treatment is covered up to ₹10,000 per policy year."
  ],
  "documents_with_content": 1,
  "processing_time": 4.218,
  "cache_hit": false
}
```

| Field | Type | Description |
|---|---|---|
| `answers` | `string[]` | One answer per question, same order |
| `documents_with_content` | `int` | Number of documents that yielded extractable text |
| `processing_time` | `float` | Wall-clock seconds for the entire request |
| `cache_hit` | `bool` | `true` if FAISS indexes were reused from cache |

### Low-confidence fallback

If an answer contains the phrase "not specify" or is empty, the server automatically retries with `k=15` (wider retrieval, deduped with `dict.fromkeys`) before returning.

### cURL example

```bash
curl -X POST http://localhost:8000/query \
  -H "Content-Type: application/json" \
  -d '{
    "documents": ["https://example.com/policy.pdf"],
    "questions": ["Is physiotherapy covered?"]
  }'
```

### Python example

```python
import requests

resp = requests.post("http://localhost:8000/query", json={
    "documents": ["https://example.com/policy.pdf"],
    "questions": [
        "Is cataract surgery covered for 10L sum insured?",
        "What are the room rent limits?",
    ]
})
data = resp.json()
for ans in data["answers"]:
    print(ans)
```

## GET /stream-query

Streams the answer as Server-Sent Events (SSE). Tokens appear as they are generated by Gemini. The endpoint also streams progress messages for each pipeline stage.

**Best for:** the web UI, chat interfaces, anywhere progressive rendering improves UX.

### Query parameters

| Param | Type | Required | Description |
|---|---|---|---|
| `question` | `string` | ✅ | The question to answer |
| `documents` | `string` | ✅ | Comma-separated PDF URLs |

### Event stream format

Each line is a `data:` SSE event carrying a JSON object:

```
data: {"message": "⚡ Using cached document index…"}

data: {"message": "🤖 Generating answer…"}

data: {"token": "Cataract surgery"}

data: {"token": " is covered"}

data: {"token": " up to ₹25,000."}

data: {"answer": "Cataract surgery is covered up to ₹25,000.", "cache_hit": true}
```

On a cache miss the stream emits:

```
data: {"message": "📄 Loading documents…"}

data: {"message": "🔢 Vector index ready."}

data: {"message": "🤖 Generating answer…"}
```

| Event key | Description |
|---|---|
| `message` | Pipeline progress update |
| `token` | One streamed text fragment from Gemini |
| `answer` | Full concatenated answer (final event) |
| `cache_hit` | Boolean on the `answer` event |
| `error` | Error string if anything failed |

### JavaScript (EventSource) example

```javascript
const params = new URLSearchParams({
  question: "Is cataract surgery covered?",
  documents: "https://example.com/policy.pdf",
});

const sse = new EventSource(`/stream-query?${params}`);
let answer = "";

sse.onmessage = (e) => {
  const data = JSON.parse(e.data);

  if (data.token) {
    answer += data.token;
    document.getElementById("output").textContent = answer;
  }

  if (data.answer || data.error) {
    sse.close();
  }
};
```

### cURL example

```bash
curl -N "http://localhost:8000/stream-query?\
question=Is+cataract+covered&\
documents=https://example.com/policy.pdf"
```

## GET /cache-status

Inspect what is currently in the LRU document cache.

```bash
curl http://localhost:8000/cache-status
```

```json
{
  "cached_document_sets": 1,
  "entries": [
    {
      "key": "a3f9b2c1…",
      "documents": ["https://example.com/policy.pdf"],
      "hits": 7,
      "age_seconds": 312
    }
  ]
}
```

`hits` is the number of times a cached index has been reused since it was built. Cache capacity is 10 entries; least-recently-used is evicted when full.

---

## POST /reset

Clears all cached document indexes. The next query for any document set will run the full pipeline again.

```bash
curl -X POST http://localhost:8000/reset
```

```json
{ "status": "reset", "entries_cleared": 2 }
```

---

## Error responses

| HTTP status | Cause |
|---|---|
| `400` | No documents, no questions, or all documents failed extraction |
| `422` | Malformed JSON body (Pydantic validation error) |
| `500` | Unexpected server error — check application logs |
